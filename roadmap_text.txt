Introduction & Strategy

Introduction & Strategy

This roadmap provides a detailed, step-by-step technical guide for transforming the current Phase
0 data pipeline into a production-grade operational intelligence platform. The strategy is built on a
core insight from our architecture comparison: the TATA Motors team has built the best-in-class
data collection engine, and the Naxis team has built the right platform architecture. The path
forward is not to choose one over the other, but to integrate them into a single cohesive system.
The roadmap spans 24 weeks (6 months), organized into 6 phases from Phase 0 (current state)
through Phase 5 (production deployment). Each phase builds incrementally on the previous one,
ensuring that value is delivered at every milestone and that the architecture remains stable and
testable throughout.
The guiding principles for this roadmap are: (1) Preserve what works -- the BasePoller, retry logic,
mock mode, and 12-field schema are production-ready and should be kept intact; (2) Layer on
capabilities -- each phase adds a new layer (persistence, API, UI, intelligence, production) without
breaking existing functionality; (3) Deliver value early -- by Phase 3 (week 10), the platform should
have a working UI that network operators can actually use; (4) Plan for scale -- every technical
decision in early phases should not prevent horizontal scaling, multi-tenancy, or advanced
analytics in later phases.

Step 3: Modify BasePoller to Write to PostgreSQL

The critical change in Phase 1 is modifying the _save_raw method and adding a new
_persist_devices method. The raw responses should continue to be saved as JSON files (for the
immutable audit trail) but should also be inserted into the raw_responses table. The normalized
devices should be upserted into the devices table using SQLAlchemy's on_conflict_do_update
equivalent.

   # Database connection from environment

   def _persist_devices(self, devices: List[Dict[str, Any]]) -> int:
         """Upsert normalized devices into PostgreSQL."""
         session = self.Session()
         count = 0
         try:
               for device_dict in devices:

                           platform=self.platform,
                           device_id=device_dict.get("device_id"),
                           hostname=device_dict.get("hostname"),
                           ip_address=device_dict.get("ip_address"),

                           raw_data=device_dict.get("_raw"),
                     )
                     session.merge(device) # Upsert on conflict
                     count += 1
               session.commit()
               logger.info("Persisted %d %s devices to PostgreSQL", count, self.platform)
         except Exception as exc:
               session.rollback()
               logger.error("Failed to persist devices: %s", exc)
               raise
         finally:
               session.close()
         return count

2.1 Dual-Schema Architecture

Phase 2 introduces the concept of a dual-schema architecture that supports both device inventory
(from the TATA Motors 12-field schema) and event-driven telemetry (from the Naxis UnifiedEvent
schema). This is essential because a complete operational platform needs both: device inventory
provides the topology context for understanding where events occur, and event streams provide
the operational signals that drive incident detection.

The device schema (already defined in Phase 1) is inventory-centric -- it answers "what devices
exist and what are their static properties?" The event schema is new in this phase and answers
"what is happening in the network right now?" Events are time-series data: alarms, state changes,
metric thresholds exceeded, authentication failures, and so on.

         device_id = Column(Integer, ForeignKey("devices.id"), nullable=True)
         platform = Column(String(20), nullable=False, index=True)
         severity = Column(String(20), default="info") # critical, warning, info
         message = Column(Text)
         event_data = Column(JSON) # Vendor-specific event payload
         source_ip = Column(String(45))
         site = Column(String(255), index=True)
         timestamp = Column(DateTime(timezone=True), nullable=False, index=True)
         correlated = Column(Boolean, default=False) # Has this been processed?
         created_at = Column(DateTime(timezone=True), server_default=func.now())

         # Relationship
         device = relationship("Device", back_populates="events")

         __table_args__ = (
               Index('ix_events_site_time', 'site', 'timestamp'),
               Index('ix_events_platform_time', 'platform', 'timestamp'),

         )

   # Add to Device class:
   events = relationship("NetworkEvent", back_populates="device")

The correlated flag is a simple but important mechanism: when the correlation engine (Phase 4)
processes an event, it sets this flag to True . This prevents duplicate processing and allows the
system to track which events have been incorporated into incidents. The device_id foreign key
links events to the device inventory, enabling topology-aware correlation -- if an event occurs at
Site A, the system can quickly find all devices at Site A and assess the blast radius.

3.2 Key Pages & Components

The three most important pages are the Incident Dashboard (main landing page), the Device
Explorer (for browsing the network inventory), and the Incident Detail Page (for investigating
specific issues). Each page is designed to provide network operators with the information they
need to understand network health and respond to incidents.

The Incident Dashboard ( / ) displays a list of active incidents with real-time auto-refresh every 10
seconds. It uses TanStack Query's useQuery hook with a refetchInterval of 10000ms to achieve
this. Each incident is rendered as an IncidentCard component showing severity (color-coded),
affected site, device count, time since creation, and a brief summary.

The Device Explorer ( /devices ) provides a searchable, filterable, and sortable table of all devices
across all platforms. It supports filtering by platform (Cisco, Mist, Velo), site, reachability status,
and software version. This page directly consumes the /devices API endpoint with query
parameters for filtering.

The AI RCA service uses a Large Language Model (LLM) to analyze incidents and suggest probable

by providing a starting hypothesis that the network operator can validate.

   import openai
   from typing import List, Dict

5.2 Monitoring, Alerting & Authentication

Production readiness requires three additional systems: monitoring (to observe platform health),
alerting (to notify when things go wrong), and authentication (to control access).
Monitoring is implemented with Prometheus for metrics collection and Grafana for visualization.
Key metrics include: API response times (p50, p95, p99), database query latency, worker queue
depth (Celery), polling success/failure rates per vendor, incident correlation accuracy, and
frontend page load times. These metrics are exposed through Prometheus endpoints on each
service.
Alerting uses Prometheus Alertmanager to send notifications (Slack, PagerDuty, email) when
thresholds are breached. Critical alerts include: polling failure rate above 10%, database
connection pool exhausted, API error rate above 1%, and worker queue depth growing without
bound.
Authentication uses Keycloak (or Azure AD B2C) for OAuth2/OIDC-based authentication. This
provides SSO integration with TATA Motors' existing identity infrastructure, role-based access
control (RBAC) for different operator levels (viewer, operator, admin), and audit logging of all user
actions.

   from fastapi import Depends, HTTPException
   from fastapi.security import OAuth2PasswordBearer
   import jwt

   oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")

   async def get_current_user(token: str = Depends(oauth2_scheme)) -> User:
         """Validate JWT token from Keycloak and return user."""
         try:
               payload = jwt.decode(
                     token,
                     key=KEYCLOAK_PUBLIC_KEY,
                     algorithms=["RS256"],
                     audience="network-platform",
               )
               return User(
                     id=payload["sub"],
                     email=payload["email"],
                     name=payload["name"],
                     roles=payload.get("realm_access", {}).get("roles", []),
               )
         except jwt.InvalidTokenError:
               raise HTTPException(status_code=401, detail="Invalid token")
